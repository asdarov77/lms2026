Проанализируй и доработай техническое задание (ТЗ) для LMS-проекта на стеке Vuetify (Vue 3) + Laravel (PHP) + PostgreSQL. Задача — подготовить готовое к передаче разработчику, полнофункциональное ТЗ с дизайном всех модулей и полной API-спекой.

Требования к результату (выдать в структурированном виде):

Краткое резюме текущего ТЗ — что включено, что отсутствует, ключевые риски и пробелы.
Список всех бизнес-требований и пользовательских ролей (Admin, Instructor, Student, Guest, Support и т.д.) с разрешениями (ACL).
Полный перечень модулей и их функций (минимум: авторизация/регистрация, профиль, каталог курсов, курс-ленты/уроки, тесты/викторины, прогресс/сертификаты, платежи, уведомления, отчёты, админ-панель, интеграции).
Для каждого модуля:
пользовательские истории (3–6 ключевых сценариев на роль),
подробный список экранов/страниц и компонентов интерфейса (включая поля форм, валидации, состояния — loading/empty/error),
требования к UX (поведенческие кейсы: навигация, модальные окна, inline-edit, drag&drop для курсов и материалов и т.п.),
доступность (a11y) и адаптивность (mobile/tablet/desktop).
Дизайн: подробное текстовое описание визуальной системы для всех модулей (цветовая схема, типографика, отступы, сетка), плюс описание ключевых экранов (доска студента, просмотр урока, создание курса, админка). Для каждого ключевого экрана укажи:
структура (какие блоки и в каком порядке),
поведение компонентов (кнопки, карточки, табы, фильтры),
примеры скрытых/появляющихся состояний (ошибки, успех, подтверждение удаления). Примечание: макеты в графическом виде не требуются — достаточно детализированного текстового дизайна, пригодного для быстрой верстки в Vuetify.
Техническая архитектура:
структура БД (основные сущности и связи — ERD в виде таблиц: поля, типы, индексы, внешние ключи, ограничения уникальности),
backend: рекомендуемые Laravel-пакеты (версия Laravel, использование Sanctum/Passport, Queue, Notifications, Policies, Resource/Transformer),
frontend: структура проекта на Vue 3 + Vuetify (страницы, компоненты, store/pinia, маршрутизация, авторизация),
CI/CD, тестирование (unit, feature, e2e), миграции и наполнители (seeders).
API-спецификация (OpenAPI-совместимая): полный набор REST-эндпоинтов для всех сущностей, с указанием:
путь и версия (/api/v1/...),
метод (GET/POST/PUT/PATCH/DELETE),
описание назначения,
параметры запроса (path, query, body) с примерами и валидацией,
образцы ответов (успех, ошибки с кодами) и HTTP-коды,
требования к аутентификации и правам доступа,
пагинация, фильтры, сортировка,
вебхуки и события (например, оплата, завершение курса).
Безопасность и масштабирование:
меры по безопасности (rate limiting, input validation, CSRF, XSS, защита файлов, шифрование чувствительных данных),
рекомендации по нагрузке (indexing, кеширование, балансировка, разделение read/write для Postgres, бэкап).
Интеграции:
платежные провайдеры (Stripe/PayPal/местные, вебхуки),
почтовые сервисы (SMTP, SendGrid, SES),
аналитика (Google Analytics/GA4, event-tracking),
SSO/OAuth (опционально).
Нефункциональные требования: SLA, время отклика API, требования к задержкам для стриминга видео (если используется), лимиты хранения, GDPR/локальные требования к хранению данных (если релевантно).
План релиза и MVP: предложи минимальную версию для запуска (список обязательных фич), и дорожную карту по приоритетам (3 релиза).
Техническое задание на развёртывание: окружения (dev/stage/prod), docker-compose/образы, конфигурации для PostgreSQL, миграций, резервного копирования, инструкции по деплою.
Список deliverables для передачи разработчику (файлы, схемы, API-спецификация в OpenAPI YAML/JSON, ERD, тексты писем, seed-данные).
Оценочные трудозатраты (часы) и уровни задач (frontend/backend/QA/DevOps) — ориентировочно для MVP и полного релиза.
Риски и откатные планы: возможные технические и организационные риски и рекомендации по их снижению.
Требования к стилю и формату выдачи:

Выдай ТЗ на русском языке в структурированном виде с заголовками и списками. Используй таблицы для сравнения вариантов и для расписаний/roadmap.
API-спеку подай также в виде таблиц и/или примерного OpenAPI-фрагмента (yaml/json) для ключевых эндпоинтов (аутентификация, курсы, уроки, тесты, платежи).
Всегда используй маршруты с версионированием (/api/v1/...).
Сделай разумные предположения по неуказанным деталям (например, форматы медиа, ограничения размера файлов, политики сроков хранения) и кратко пометь их как «предположение».
Не задавай уточняющих вопросов — когда что-то неоднозначно, выбери разумный вариант и укажи, что это предположение.
Технические ограничения/предпочтения:

Laravel 10+ или выше, PHP 8.1+;
Vue 3 + Vuetify 3 (composition API);
PostgreSQL 13+;
Аутентификация: Laravel Sanctum (предпочтительно) — если предлагаешь альтернативы, опиши плюсы/минусы в таблице;
Хранение медиа: локально + S3-совместимое хранилище (предположение);
CI: GitHub Actions (предположение).
Выходные файлы:

Сформируй финальный документ ТЗ (текст), плюс экспорт ключевых частей: OpenAPI-фрагмент (yaml) и ERD в виде таблиц. Пометь явные предположения.
Сроки: подготовь полный документ (MVP + API + дизайн-описание) — оценочно 5–7 рабочих дней; если считаешь по-другому, укажи свою оценку в часах/днях в конце.
